Account and access support

  1. Confirm the person and approved email through DPIK's normal channel.
  2. Check whether the identity and active Project membership exist; do not bypass RLS with browser-side service credentials.
  3. Grant only the required capability bundle with a recorded reason and granting identity.
  4. For revocation, deactivate the membership/grant and verify that a fresh authenticated session is denied.
  5. Record the change and its verification evidence.

The application does not yet provide a complete self-service administration screen. Database access changes must follow a reviewed migration or approved administrative procedure—never improvised SQL against the shared project.

Daily health check

  1. Run Start-TUGAS.cmd and confirm the local health response.
  2. Sign in as an approved fixture user and confirm Project context and My Work load.
  3. Check that anonymous access remains denied.
  4. Escalate migration mismatch, repeated stale-version errors, missing Project context, or outbox/projection lag.

Credential incidents

If a token, key, magic link, or signing key is exposed: stop using it, revoke/rotate it at the provider, update the approved secret store, prove the previous value no longer works, and record credential-free evidence. Never paste the replacement into chat, issues, logs, or documentation.
